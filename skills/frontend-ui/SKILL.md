# Frontend UI

## Overview

Build production-quality user interfaces that are accessible, performant, and visually polished. Output should look like it was built by a design-aware engineer at a top company — not generated by an AI. That means real design system adherence, proper accessibility, thoughtful interaction patterns, and zero generic "AI aesthetic."

## The Iron Law

```
EVERY INTERACTIVE ELEMENT IS KEYBOARD ACCESSIBLE
COLOR IS NEVER THE SOLE INDICATOR OF STATE
```

A `<div onClick>` is not a button. A red border without an icon or message is not an error state. If a sighted keyboard user or a screen reader user cannot complete the flow, the work is unfinished — not "polished later."

## When to Use

- Building new UI components or pages
- Modifying existing user-facing interfaces
- Implementing responsive layouts
- Adding interactivity or state management
- Fixing visual or UX issues
- Reviewing AI-generated UI before shipping it

## The AI Aesthetic Anti-Patterns

The defining failure mode. AI-generated UI has a recognizable look that signals low quality. **Refuse these defaults. Always anchor to the project's actual design system.**

| AI Default | Why It's a Problem | Production Quality |
|---|---|---|
| Purple/indigo everything | Models default to "safe" palettes, making every app look identical | Use the project's actual color palette; check `tailwind.config`, CSS custom properties, or the design system |
| Excessive gradients | Add visual noise, clash with most design systems, signal "demo" not "product" | Flat surfaces, or subtle gradients only when the design system specifies them |
| Rounded everything (rounded-2xl) | Maximum rounding signals "friendly" but ignores the corner-radius hierarchy in real designs (cards ≠ buttons ≠ inputs) | Consistent border-radius pulled from the design system tokens |
| Generic hero sections | Centered headline + subhead + CTA template, disconnected from actual content or user need | Content-first layout driven by what the user is here to do |
| Lorem-ipsum copy | Placeholder text hides layout problems real content reveals (length, wrapping, overflow, line-breaks at narrow widths) | Realistic copy at realistic lengths; ask the user for it if missing |
| Oversized padding everywhere | Equal generous padding destroys visual hierarchy and wastes screen space — especially mobile | Spacing scale: tighter for related content, looser for separation |
| Stock card grids | Uniform 3- or 4-column grids ignore information priority and scanning patterns | Purpose-driven layout: featured/secondary, list/detail, master/feed |
| Shadow-heavy design | Layered shadows compete with content and slow rendering on low-end devices | Subtle shadow or none unless the design system specifies elevation |

**When the user says "make it look modern":** ask which design system, palette, or reference site to follow. Do not invent a purple-gradient hero.

## Component Architecture

### File Colocation

Group related files by component, not by type:

```
src/components/
  TaskList/
    TaskList.tsx          # Component
    TaskList.test.tsx     # Tests
    TaskList.stories.tsx  # Storybook (if used)
    use-task-list.ts      # Hook (if complex state)
    types.ts              # Component-specific types
```

### Composition Over Configuration

Smaller, focused pieces beat heavily-parameterized mega-components.

```tsx
// Good: composable
<Card>
  <CardHeader><CardTitle>Tasks</CardTitle></CardHeader>
  <CardBody><TaskList tasks={tasks} /></CardBody>
</Card>

// Avoid: config soup
<Card title="Tasks" headerVariant="large" bodyPadding="md" content={<TaskList tasks={tasks} />} />
```

### Container / Presentation Split

Keep data-fetching out of rendering. Fetch state lives in containers; presentation components receive plain props.

```tsx
// Container: handles data + states
export function TaskListContainer() {
  const { tasks, isLoading, error, refetch } = useTasks();
  if (isLoading) return <TaskListSkeleton />;
  if (error)     return <ErrorState message="Failed to load tasks" onRetry={refetch} />;
  if (!tasks.length) return <EmptyState message="No tasks yet" />;
  return <TaskList tasks={tasks} />;
}

// Presentation: pure rendering
export function TaskList({ tasks }: { tasks: Task[] }) {
  return (
    <ul role="list" className="divide-y">
      {tasks.map(t => <TaskItem key={t.id} task={t} />)}
    </ul>
  );
}
```

## State Management

Choose the simplest tool that works. Walk down this tree, not up:

```
Local state (useState)            → Component-specific UI state (open/closed, hover)
Lifted state                      → Shared between 2-3 sibling components
Context                           → Theme, auth, locale (read-heavy, write-rare)
URL state (searchParams)          → Filters, pagination, shareable UI state
Server state (React Query, SWR)   → Remote data with caching/invalidation
Global store (Zustand, Redux)     → Complex client state shared app-wide
```

**Avoid prop drilling deeper than 3 levels.** If you're passing props through components that don't use them, introduce context or restructure the tree. Don't reach for a global store before you've tried the smaller options.

## Design System Adherence

### Spacing Scale

Use the project's scale. Don't invent values.

```css
/* Good — on the scale */
padding: 1rem;     /* 16px */
gap: 0.75rem;      /* 12px */

/* Bad — arbitrary, signals AI */
padding: 13px;
margin-top: 2.3rem;
```

### Typography Hierarchy

```
h1 → Page title (one per page)
h2 → Section title
h3 → Subsection title
body → Default text
small → Secondary/helper text
```

Don't skip heading levels. Don't style non-headings as headings (or vice versa) — screen readers use heading structure to navigate.

### Semantic Color Tokens

- Use tokens: `text-primary`, `bg-surface`, `border-default` — never raw hex in components
- Contrast ≥ 4.5:1 for body text, ≥ 3:1 for large text and UI controls
- Color is never the sole signal — pair with icon, text, or pattern

## Accessibility (WCAG 2.1 AA)

Every component meets these. Heavy reference material lives in `accessibility-checklist.md` (this file is the summary).

### Keyboard Navigation

```tsx
// Always
<button onClick={handleClick}>Click me</button>           // focusable, Enter/Space work

// Never
<div onClick={handleClick}>Click me</div>                 // not focusable, no keyboard

// Only if you have to (custom widget):
<div role="button" tabIndex={0}
     onClick={handleClick}
     onKeyDown={e => { if (e.key === 'Enter' || e.key === ' ') { e.preventDefault(); handleClick(); } }}>
  Click me
</div>
```

If your reflex is `<div onClick>`, stop and use `<button>` (action) or `<a href>` (navigation).

### ARIA Labels

```tsx
<button aria-label="Close dialog"><XIcon /></button>      // icon-only buttons
<label htmlFor="email">Email</label><input id="email" />  // form inputs
<input aria-label="Search tasks" type="search" />         // when no visible label
```

### Focus Management

Move focus when content changes. Trap focus in modals. Return focus to the trigger on close.

```tsx
function Dialog({ isOpen, onClose }: DialogProps) {
  const closeRef = useRef<HTMLButtonElement>(null);
  useEffect(() => { if (isOpen) closeRef.current?.focus(); }, [isOpen]);
  return (
    <dialog open={isOpen} aria-modal="true">
      <button ref={closeRef} onClick={onClose}>Close</button>
      {/* content */}
    </dialog>
  );
}
```

### Meaningful Empty and Error States

Blank screens are bugs. Every list, query, and form has three states beyond success: loading, empty, error.

```tsx
if (tasks.length === 0) {
  return (
    <div role="status" className="text-center py-12">
      <TasksEmptyIcon className="mx-auto h-12 w-12 text-muted" />
      <h3 className="mt-2 text-sm font-medium">No tasks</h3>
      <p className="mt-1 text-sm text-muted">Get started by creating a new task.</p>
      <Button className="mt-4" onClick={onCreateTask}>Create Task</Button>
    </div>
  );
}
```

For deeper checks (live regions, focus traps, common HTML patterns, testing tools), see the checklist file in this skill directory.

## Responsive Design

Mobile-first. Test at 320px, 768px, 1024px, 1440px.

```tsx
<div className="
  grid grid-cols-1      /* mobile: single column */
  sm:grid-cols-2        /* ≥640: two */
  lg:grid-cols-3        /* ≥1024: three */
  gap-4
">
```

Touch targets ≥ 44×44px on mobile. Don't make desktop-first then "make it responsive later" — retrofit is 3x harder.

## Loading and Transitions

Skeletons, not spinners, for content. Optimistic updates for fast perceived response.

```tsx
function TaskListSkeleton() {
  return (
    <div className="space-y-3" aria-busy="true" aria-label="Loading tasks">
      {Array.from({ length: 3 }).map((_, i) => (
        <div key={i} className="h-12 bg-muted animate-pulse rounded" />
      ))}
    </div>
  );
}

// Optimistic update with rollback on error
function useToggleTask() {
  const qc = useQueryClient();
  return useMutation({
    mutationFn: toggleTask,
    onMutate: async (taskId) => {
      await qc.cancelQueries({ queryKey: ['tasks'] });
      const previous = qc.getQueryData<Task[]>(['tasks']);
      qc.setQueryData<Task[]>(['tasks'], old =>
        old?.map(t => t.id === taskId ? { ...t, done: !t.done } : t) ?? []);
      return { previous };
    },
    onError: (_e, _id, ctx) => qc.setQueryData(['tasks'], ctx?.previous),
  });
}
```

## Common Rationalizations

| Rationalization | Reality |
|---|---|
| "Accessibility is a nice-to-have" | Legal requirement in many jurisdictions; engineering quality standard everywhere. |
| "We'll make it responsive later" | Retrofit is 3x harder. Mobile-first now. |
| "Design isn't final, skip styling" | Use design system defaults. Unstyled UI breaks the first impression. |
| "The AI aesthetic is fine for now" | Signals low quality. Use the project's actual design system from the start. |
| "Just a prototype" | Prototypes become production. Build the foundation right. |
| "Make it modern — purple gradient hero is fine" | "Modern" without a reference is AI-default. Ask which design system or palette to follow. |
| "A `<div onClick>` is simpler than a button" | It's broken for keyboard users and screen readers. `<button>` is the simpler thing. |
| "I'll add the empty/error state once it works" | Real users hit those branches first. They are the feature, not polish. |
| "I don't have real copy yet" | Ask. Lorem ipsum hides every layout bug that real content exposes. |

## Red Flags

- Components over 200 lines (split them)
- Inline styles or arbitrary pixel values (`padding: 13px`, `margin-top: 2.3rem`)
- Missing loading, empty, or error states
- No keyboard navigation testing (you didn't Tab through it once)
- Color as sole signal (red border without icon or message)
- Generic "AI look": purple/indigo, oversized rounded-2xl cards, shadow-heavy, lorem ipsum, stock card grids
- `<div onClick>` or `<span onClick>` for actions
- Heading levels skipped (h1 → h3) or used for non-heading visual styling
- Hex colors written directly in components instead of tokens

## Verification

Before considering UI work complete:

- [ ] Component renders without console errors or warnings
- [ ] Every interactive element reachable and operable via keyboard (Tab, Enter, Space, Escape)
- [ ] Visible focus indicator on all focusable elements
- [ ] Screen reader conveys structure and content (headings, labels, live regions for dynamic updates)
- [ ] Works at 320px, 768px, 1024px, 1440px
- [ ] Loading, empty, and error states all implemented (not just success)
- [ ] Spacing, color, typography pulled from the project's design system — no raw hex, no arbitrary px
- [ ] Color is never the only signal of state
- [ ] No accessibility violations in dev tools (axe-core, Lighthouse Accessibility)
- [ ] Real or realistic copy in place — no lorem ipsum left behind
